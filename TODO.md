### Bugs Fixed (14 total)

1. **ViT layernorm ordering**: pre-norm (ln before attention), not post-norm
2. **Absolute positional embedding**: was missing entirely (pos_embed)
3. **RoPE frequency computation**: wrong formula for per-block tables
4. **Metal GELU**: MLX GELU didn't match PyTorch's tanh approximation
5. **Encoder V-without-pos**: V=norm(x), not V=norm(x)+pos
6. **BPE vocabulary loading**: vocab file parsing errors
7. **Geometry encoder rewrite**: CPU-based 3-layer transformer
8. **Prompt token encoding**: coordinate normalization + label embedding
9. **BPE tokenization**: encoding correctness fixes
10. **Decoder box refinement**: iterative per-layer (not batched)
11. **Geometry encoder CLS token order**: CLS prepended, not appended
12. **Text padding mask**: truncate text features to real tokens only
    (Python masks padding in encoder/decoder cross-attention;
    without truncation, ~90% of attention goes to padding tokens)
13. **Image resize: letterbox → direct squash**: C was letterboxing
    (aspect-preserving resize + black padding) instead of squash-resizing
    to match Python's `v2.Resize(size=(1008, 1008))`
14. **MLX contiguous copy in Phase 3**: Metal backend copied non-contiguous
    MLX array data (transpose/reshape views) directly via memcpy, getting
    the pre-transpose memory layout instead of the logical layout. Fixed by
    adding `mlx_contiguous()` before copy. This was the root cause of the
    ViT patch embedding corruption (conv2d NHWC→NCHW not materialized).

### Remaining Divergence Analysis

The remaining pred_masks divergence (max=101) is **not from code bugs**.
All intermediate stages through the segmentation head are now EXACT
(max < 0.003), so the divergence originates in the decoder's dot-product
scoring (200 queries × 256d × 288² pixels), which amplifies small
floating-point differences to max=101.

Evidence:
- Neck: max=0.0025 (Metal FP32 baseline)
- Encoder L0–L5: max=0.0026 (EXACT — no amplification through encoder)
- Seg head pixel/instance: max=0.0005 (EXACT)
- Dot product scoring amplifies to max=101

The C encoder value ranges match Python closely:
  L5: C [-22.59, 12.25] vs Py [-22.68, 12.57]
The error is in the tail of the distribution, not systematic bias.

When running on real images (not fixture-normalized input), an additional
source of difference is the image resize algorithm (stb_image_resize2 vs
torchvision bilinear). This adds max=0.25 pixel-level input difference
which amplifies through the 32 ViT blocks but does not affect model
correctness — segmentation output is visually correct.

No further code bugs were found after verifying:
- Encoder: pre-norm, pos_enc_at_attn=True, cross_attn_keys/queries=False
- Decoder: post-norm, SA→TCA→ICA→FFN, relu activation, box refinement
- Decoder substeps (SA, TCA, CA): EXACT on same inputs (max=1e-5)
- SDPA scaling: 1/√32 (correct)
- Position encoding: numerically identical (max diff 6e-7)
- Text features: numerically identical (max diff 6e-6)
- Weight QKV fusing: verified correct splits
- Conv2d: EXACT vs PyTorch on same input (max=1e-5 after contiguous fix)

## Quick Reference — API Usage

### Image Inference
```python
from sam3.model_builder import build_sam3_image_model
from sam3.model.sam3_image_processor import Sam3Processor

model = build_sam3_image_model()
processor = Sam3Processor(model)
state = processor.set_image(image)
state = processor.set_text_prompt("your prompt", state)
masks, boxes, scores = state["masks"], state["boxes"], state["scores"]
```

### Video Inference (SAM 3.1)
```python
from sam3.model_builder import build_sam3_multiplex_video_predictor

predictor = build_sam3_multiplex_video_predictor()
resp = predictor.handle_request({"type": "start_session", "resource_path": video_path})
predictor.handle_request({"type": "add_prompt", "session_id": resp["session_id"], "frame_index": 0, "text": "person"})
for out in predictor.handle_stream_request({"type": "propagate_in_video", "session_id": resp["session_id"]}):
    frame_masks = out["outputs"]
```
